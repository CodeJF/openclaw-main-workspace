# 2026-03-11

- Spent substantial time debugging OpenClaw cron delivery/execution for a daily Shenzhen weather Feishu reminder.
- Confirmed multiple separate issues during setup:
  - Feishu pairing/access temporarily broke and was restored by approving pairing code `WTB9UJU7`.
  - Main session store had many stale/broken session references; `openclaw sessions cleanup --all-agents --enforce --fix-missing` reduced main store from 11 entries to 2 valid ones.
  - Gateway service was normalized with `openclaw gateway install --force`; after that, agent/configs remained intact.
- Key technical finding: OpenClaw cron jobs were being enqueued and marked with `runningAtMs`, but no `lastRunAtMs` / `lastStatus` / run history entries were produced.
- Minimal repro showed the problem is not specific to the weather script or Feishu delivery:
  - a minimal `main` `systemEvent` cron job also got stuck,
  - a minimal `isolated` `agentTurn` cron job also got stuck.
- Most likely root cause identified from local OpenClaw code + state inspection:
  - cron marks due jobs as `running` before completion,
  - default cron concurrency is 1,
  - `isolated agentTurn` cron jobs have a long default safety timeout (60 min),
  - a first stuck isolated job can monopolize the cron lane and make later jobs appear running without finishing.
- Suggested next remediation path: set `cron.maxConcurrentRuns` > 1 (e.g. 2) and recreate the weather cron with an explicit short `timeoutSeconds` (e.g. 60s), then clear stale running state and retest.
